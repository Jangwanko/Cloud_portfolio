이 데모는 단순한 채팅 CRUD가 아니라, 메시지 API와 비동기 worker를 분리하고, 중복 방지와 운영 관측까지 함께 설명할 수 있도록 구성한 메시징 시스템 포트폴리오입니다.

API / Worker 분리 사용자 응답 경로와 후속 처리를 나눠 운영 확장성을 고려했습니다.
중복 요청 방지 `X-Idempotency-Key`로 모바일 재시도 상황을 설명할 수 있습니다.
운영 시야 Observer에서 send와 worker 처리 타임라인을 함께 볼 수 있습니다.
HA 확장 경로 Kubernetes에서 PostgreSQL HA와 Redis Sentinel로 확장 가능합니다.

메시지 흐름

면접 시에는 아래 순서대로 설명하면 구조적 사고가 더 잘 드러납니다.

1
API가 메시지를 저장합니다. 메시지는 PostgreSQL에 기록되고, 사용자 응답 경로는 우선 빠르게 끝냅니다.
2
후속 작업은 Redis queue로 분리합니다. 알림, fan-out, 비동기 후처리는 worker로 넘겨 API와 결합도를 낮춥니다.
3
Worker가 처리 이력을 남깁니다. `notification_attempts`에 기록해 "전송"과 "처리"를 구분해서 볼 수 있습니다.
4
Observer로 운영 흐름을 검증합니다. 기능 구현을 넘어서 운영 관점까지 보여주는 것이 이 포트폴리오의 차별점입니다.

현재 강점과 다음 단계

좋은 포트폴리오는 완성형보다 방향성이 분명해야 합니다.

강점: 면접 시연이 빠릅니다. `docker compose`로 바로 실행되고, API와 관측 화면을 함께 보여줄 수 있습니다.
강점: 운영 친화적인 설명이 가능합니다. idempotency, readiness, worker 분리, Redis Sentinel 확장 포인트를 자연스럽게 연결할 수 있습니다.
다음 단계: 메트릭과 장애 대응을 보강해야 합니다. 처리량, queue 지연, 재시도, DLQ, 실패율이 추가되면 포트폴리오 무게감이 크게 올라갑니다.
현재 한계: 인증과 실시간 push는 아직 없습니다. 이 부분은 숨기기보다 "왜 아직 넣지 않았고 다음에 어떻게 확장할지"를 명확히 말하는 편이 좋습니다.
